# Codebase Concerns

**Analysis Date:** 2026-03-24

## Tech Debt

**Scaffold-only codebase — no real application code:**
- Issue: Both `apps/api` and `apps/web` are untouched scaffolds. The API exposes a single `GET /` returning `"Hello World!"`. The web app renders the default Next.js "create-next-app" starter page with Vercel promotional links.
- Files: `apps/api/src/app.controller.ts`, `apps/api/src/app.service.ts`, `apps/web/src/app/page.tsx`
- Impact: There is no production-ready functionality. Every feature of the intended product still needs to be built from scratch.
- Fix approach: Replace scaffold handlers and page content with actual domain logic as implementation phases proceed.

**`packages/` directory is empty:**
- Issue: The monorepo workspace declares `"packages/*"` in the root `package.json`, but the `packages/` directory contains no packages. Shared types, utilities, and API contracts have nowhere to live yet.
- Files: `/home/narumir/projects/moew/package.json`, `/home/narumir/projects/moew/packages/`
- Impact: Any cross-app sharing (e.g., shared TypeScript types between API and web) currently has no home, which encourages duplication or ad-hoc imports.
- Fix approach: Create at least one shared package (e.g., `packages/types`) as soon as a contract between API and web is needed.

**No environment configuration layer:**
- Issue: The API bootstraps with `process.env.PORT ?? 3000` inline in `apps/api/src/main.ts` but uses no NestJS `ConfigModule` or validation layer. No `.env.example` or documentation of required environment variables exists.
- Files: `apps/api/src/main.ts`, `apps/api/src/app.module.ts`
- Impact: Contributors have no reference for what env vars are required. Misconfiguration will cause silent runtime failures rather than clear startup errors.
- Fix approach: Add `@nestjs/config` with `ConfigModule.forRoot({ validate: ... })` using a schema validation library (e.g., `zod` or `class-validator`). Add a `.env.example` file at the repo root documenting all required variables.

**Web app metadata is generic scaffold copy:**
- Issue: `apps/web/src/app/layout.tsx` exports `title: "Create Next App"` and `description: "Generated by create next app"`. These are placeholder values that will be served in production.
- Files: `apps/web/src/app/layout.tsx`
- Impact: Incorrect metadata ships to production (SEO, social sharing, browser tabs).
- Fix approach: Replace with real product name and description.

**Font usage inconsistency in CSS:**
- Issue: `apps/web/src/app/globals.css` sets `font-family: Arial, Helvetica, sans-serif` on `body`, which overrides the Geist Sans CSS variable (`--font-geist-sans`) that is loaded and injected via `layout.tsx`. The CSS variable is loaded but never applied in `globals.css`.
- Files: `apps/web/src/app/globals.css`, `apps/web/src/app/layout.tsx`
- Impact: The Google font (Geist) is downloaded but not actually used for body text, wasting bandwidth and producing an inconsistent visual result.
- Fix approach: Change the `body` rule to use `font-family: var(--font-geist-sans), Arial, Helvetica, sans-serif;`.

## Security Considerations

**No CORS configuration on API:**
- Risk: The NestJS application starts with default Express settings, meaning CORS is disabled by default. If the web frontend needs to call the API from a browser, requests will be blocked. Conversely, if CORS is later enabled permissively (e.g., `origin: '*'`), it could expose the API to unintended clients.
- Files: `apps/api/src/main.ts`
- Current mitigation: None.
- Recommendations: Add `app.enableCors({ origin: process.env.ALLOWED_ORIGINS })` with an explicit allowlist driven by environment configuration.

**No authentication or authorization layer:**
- Risk: No auth guards, JWT middleware, or session handling exist anywhere in the API.
- Files: `apps/api/src/app.module.ts`, `apps/api/src/app.controller.ts`
- Current mitigation: No routes with protected resources exist yet (scaffold only).
- Recommendations: Plan auth strategy (e.g., `@nestjs/passport` with JWT) before any protected endpoints are added.

**No input validation on API:**
- Risk: NestJS's `ValidationPipe` is not registered globally in `apps/api/src/main.ts`. Any request body or query params accepted by future controllers will not be validated.
- Files: `apps/api/src/main.ts`
- Current mitigation: No input-accepting routes exist yet.
- Recommendations: Add `app.useGlobalPipes(new ValidationPipe({ whitelist: true, forbidNonWhitelisted: true }))` in `main.ts` before any DTO-consuming routes are added.

**No HTTP security headers:**
- Risk: No Helmet or equivalent middleware is applied to the API, meaning default Express headers (e.g., `X-Powered-By: Express`) are sent and security headers (CSP, HSTS, etc.) are absent.
- Files: `apps/api/src/main.ts`
- Current mitigation: None.
- Recommendations: Add `npm install helmet` and `app.use(helmet())` in bootstrap.

## Fragile Areas

**NestJS running via Bun instead of Node:**
- Files: `apps/api/package.json` (`"test": "bun test"`, `@types/bun` devDependency)
- Why fragile: NestJS is designed and tested against Node.js. Running it under Bun can produce subtle compatibility issues with decorators, `reflect-metadata`, and Express internals. The `emitDecoratorMetadata` compiler option (required by NestJS) has known issues in Bun's TypeScript transpiler.
- Safe modification: Test all decorator-heavy NestJS features carefully. Watch for issues with `reflect-metadata` polyfill behavior.
- Test coverage: None for any decorator behavior.

**`biome.json` disables `style/useImportType` for the entire API:**
- Files: `biome.json` (overrides block for `apps/api/**`)
- Why fragile: NestJS with `emitDecoratorMetadata` requires value imports (not type-only imports) for injected dependencies. The override is correct, but it silently suppresses all import-type warnings, which could mask accidental value imports in type-only positions elsewhere in the API codebase as it grows.
- Safe modification: When importing types that are only used as types (not injected), manually add `import type` to keep intent clear.
